# JalkLiveSessionActivity ↔ JalkTimerService Lifecycle Audit

The table below catalogs every code path that touches the live session activity/service interaction, with emphasis on scenarios that can produce duplicate `JalkLiveSessionActivity` instances or reset duration fields while a session is active.

| File & Function | Finding Summary | Impact | Recommendation |
| --- | --- | --- | --- |
| `app/src/main/AndroidManifest.xml` (`<activity android:name=".JalkLiveSessionActivity">`) | Declares `android:launchMode="singleTop"`, but no task affinity/flags to prevent new task creation when started from non-activity contexts. | Intents launched without `FLAG_ACTIVITY_SINGLE_TOP`/`CLEAR_TOP` (e.g., timer alert full-screen intent) can still spawn a second instance when activity not at top. | Add consistent intent flags (or switch to `singleTask`) across every launcher, especially service-built pending intents. |
| `JalkSessionConfigActivity.launchLiveSession()` | Only entry point providing duration extras (`EXTRA_*_DURATION_MILLIS`) when user starts a session. Sets mode extra to walk by default. | Provides authoritative durations; any other launch path omits them. Subsequent relaunches lacking extras fall back to defaults. | Factor intent building into helper that other launchers reuse, or persist chosen durations for later re-entry. |
| `JalkLiveSessionActivity.onCreate()` / `extractDurationsFromIntent()` | Reads durations once from the creating intent, defaulting to 60s walk/jog & 30s rest when extras missing. No `onNewIntent` handling to refresh extras for reused instance. | If the activity is recreated by a pending intent with no extras, UI shows default durations even while service continues with real values/rep counts. | Override `onNewIntent()` to ignore missing extras, or better, query the service/shared storage for authoritative state when (re)attaching. |
| `JalkLiveSessionActivity.showIdleState()` | Called during `onCreate()` before any service broadcast arrives, resetting totals and labels to “waiting.” | When activity is recreated mid-session (screen unlock), UI temporarily looks idle until next broadcast tick. | Request an immediate state sync from service in `onStart()` (e.g., via sticky broadcast or explicit query) to avoid stale/idle gap. |
| `JalkTimerService.handleStartMode(Intent)` | Copies duration extras into in-memory fields via `updateStoredDurations()`, resets rep counters only on first mode of session. | Service preserves durations/rep counts during session, but fields vanish if service process is killed (no persistence). | Persist active-session metadata (durations, current mode, timestamps) in `SharedPreferences` or database so the service/activity can recover after restart. |
| `JalkTimerService.buildStatusNotification()` | Notification content intent targets `JalkLiveSessionActivity` with `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP`, but **omits duration extras**. | Tapping status notification while activity destroyed launches new instance that falls back to default durations even though rep counts from service continue. | Provide current durations via extras, or have activity fetch durations from service instead of intent. |
| `JalkTimerService.triggerTimerAlert()` | Full-screen pending intent uses bare `new Intent(this, JalkLiveSessionActivity.class)` (no flags/extras). | When alert fires on lock screen, Android often creates a *fresh* activity instance lacking extras; UI resets durations to defaults while service retains rep counts. | Set `FLAG_ACTIVITY_NEW_TASK | FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP` and attach stored durations/current mode extras to ensure reuse. |
| `JalkTimerService.postAlertNotification()` | Secondary alert notification (non full-screen) does set `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP`, but still no duration extras. | Same default-duration problem when user taps alert instead of status card. | Include duration extras or use shared state retrieval. |
| `JalkTimerService.handleStartMode(String,long)` (notification quick actions) | Uses `getDurationForMode()` to look up stored durations after handling quick action taps. | Keeps rep counts and durations internally accurate even when activity UI resets. | No change needed, but underscores the mismatch between service truth vs. activity intent extras. |
| `JalkTimerService.computeRemainingMillis()` & `broadcastState()` | Broadcast-only communication; activity never binds or requests snapshot. | Activity relies on periodic broadcasts; if a broadcast is missed (process pause) UI can lag or stay idle until next tick. | Consider exposing a bound service or sticky broadcast so re-created activity can immediately sync. |
| `TimerExpiredReceiver.onReceive()` | Restarts service with `ACTION_TIMER_EXPIRED` when alarm fires but passes only mode extra. | If process restarted here, service resumes with default durations because previous values were in-memory only. | Persist durations/current mode before scheduling alarm so restart path can restore them. |
| `JalkLiveSessionActivity.stopSession()` | Launches `JalkMainActivity` with `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_NEW_TASK`. | If stop invoked from notification, could create new task stack; not directly causing duplication but worth noting for task hygiene. | Consider `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP` to stay in same task. |
| Global (no singleton guard) | No central “active session” flag; `JalkSessionConfigActivity` will happily start another live session even if service already running. | Users can start overlapping sessions, spawning additional service runs or conflicting UI. | Introduce a shared guard (e.g., in `SharedPreferences` or via service query) before launching new session, and clear it when service stops. |

## Additional Notes

* `JalkTimerService` is `START_STICKY`; if the process dies and the system restarts the service with a null intent, all in-memory session fields (durations, reps, current mode) revert to defaults, yet the notification will still exist until refreshed. This can leave the activity with no accurate state to display.
* None of the pending intents bundle `TimerContract.EXTRA_MODE`/duration extras for resumption paths. Even when `singleTop` prevents duplicate instances, `onNewIntent()` is never overridden, so the activity ignores later intents that might have carried corrective data.
* There is no persistence of timer progress; only rep totals are broadcast. If the activity misses enough ticks (e.g., device asleep), it depends entirely on the next broadcast to catch up.

## Suggested Hardening Steps

1. **Centralize session state** in the service (or a `SessionManager`) persisted to disk, and have the activity fetch it on `onStart()`.
2. **Standardize intent creation** via a helper that always attaches the latest durations/mode and flags, used by ConfigActivity, notifications, and alarms.
3. **Handle `onNewIntent()`** within `JalkLiveSessionActivity` to merge incoming state rather than silently ignoring re-launches.
4. **Add duplicate-session guardrails** before starting new sessions from `JalkSessionConfigActivity`.
5. **Persist alarm recovery context** so `TimerExpiredReceiver` can restore service state accurately.
